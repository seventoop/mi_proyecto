4. MOTOR ACTUAL / FUTURO DE ANÁLISIS DE LOTES / PLANOS / IMÁGENES
================================================================================

¿Existe hoy un motor real de análisis? → PARCIALMENTE

Parser DXF:
  - Existe en lib/blueprint-utils.ts
  - Extrae entidades: LWPOLYLINE, POLYLINE, LINE, CIRCLE, TEXT
  - Convierte a SVG para visualización
  - Custom (no usa librería externa como dxf-parser)

Parser SVG:
  - Existe en lib/blueprint-utils.ts
  - Usa DOMParser nativo del browser
  - Extrae paths, polygons, rectangles

IA de visión: NO EXISTE
  - OpenAI está integrado solo para texto (leads, copilot)
  - No hay endpoints de OpenAI Vision, Claude Vision, ni Google Vision

Detección automática de lotes: NO EXISTE
  - El parser lee entidades DXF pero no crea lotes en la DB
  - Los lotes se crean manualmente via UI de manzanas/unidades

Georreferenciación automática: NO EXISTE
  - Se hace manualmente via overlay bounds en el overlay editor

Configuración manual del admin: SÍ EXISTE
  - El admin puede posicionar, escalar y rotar el plano sobre coordenadas

CONCLUSIÓN:
El sistema hoy depende de:
  1. Parsing DXF/SVG custom (conversión a SVG visual)
  2. Configuración manual del admin (posicionamiento/calibración)
  3. Creación manual de lotes (no automática desde DXF)
No hay IA de visión, no hay detección automática.

Para el futuro ideal, el flujo sería:
  1. Subir DXF → parser extrae polígonos
  2. Admin calibra sobre mapa satelital (ya existe)
  3. Sistema convierte polígonos calibrados → lotes en DB (falta)
  4. Opcionalmente: IA de visión para validar o enriquecer datos (futuro)

================================================================================
7. RIESGOS / DEUDA TÉCNICA
================================================================================

CRÍTICO:
- Sin mapa satelital: módulo más visible del producto sin librería de mapas

ALTO:
- 41+ errores TypeScript: Decimal Prisma ↔ number (compila pero es deuda real)
- proyectoId indirecto en Unidad: dificulta queries directas a escala
- Parser DXF custom sin test suite: puede fallar con DXFs complejos

MEDIO:
- TensorFlow.js client-side: bundle muy pesado para upscaling
- Storage local en dev: uploads se pierden si se limpia public/uploads/
- Sentry sin DSN: errores en producción no se capturan
- Hotspot requiere unidadId mandatory: impide hotspots sin unidad asociada

BAJO:
- Multi-tenant parcialmente integrado
- ESLint: no-unused-vars pendiente
- Sentry: deprecation warnings en sentry.server.config.ts
